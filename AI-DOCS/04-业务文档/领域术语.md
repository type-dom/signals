## 🎯 核心概念术语

### Signal (信号)

**定义**: 响应式系统的基本单元，封装可变值并自动追踪依赖。

**特征**:
- 存储单个值（任意类型）
- 通过 `get()` 读取（建立依赖）
- 通过 `set()` 更新（触发通知）
- 自动收集订阅者

**示例**:
```typescript
const count = signal(0);
count.get();  // 读取: 0
count.set(5); // 更新
```

**相关术语**: Observable (MobX), Ref (Vue), Atom (Solid)

---

### Computed (计算属性)

**定义**: 基于其他 signals 派生的只读信号，具有缓存机制。

**特征**:
- 惰性求值（访问时才计算）
- 自动缓存（依赖不变不重新计算）
- 依赖追踪（自动知道何时更新）
- 可选 setter（支持可编辑的计算值）

**示例**:
```typescript
const firstName = signal('John');
const lastName = signal('Doe');

const fullName = computed(() => {
  return `${firstName.get()} ${lastName.get()}`;
});

fullName.get(); // "John Doe" (缓存)
```

**相关术语**: Derived (MobX), Computed (Vue), Memo (Solid)

---

### Effect (副作用)

**定义**: 响应信号变化而自动重新执行的函数。

**特征**:
- 立即执行一次
- 自动追踪依赖
- 依赖变化时重新执行
- 可返回清理函数
- 可手动停止

**示例**:
```typescript
const stop = effect(() => {
  console.log('Count:', count.get());
});

count.set(5); // 自动打印: "Count: 5"

stop(); // 停止监听
```

**相关术语**: Observer (MobX), WatchEffect (Vue), CreateEffect (Solid)

---

### EffectScope (作用域)

**定义**: 管理一组 effects 的生命周期容器。

**特征**:
- 批量注册 effects
- 统一清理所有 effects
- 防止内存泄漏
- 嵌套作用域支持

**示例**:
```typescript
const scope = effectScope(() => {
  effect(() => { /* ... */ });
  effect(() => { /* ... */ });
});

scope(); // 清理所有
```

## 📊 数据结构术语

### ReactiveNode (响应式节点)

**定义**: 所有响应式对象的基类接口。

**结构**:
```typescript
interface ReactiveNode {
  deps?: Link;        // 我依赖谁
  depsTail?: Link;    // 依赖链表尾部
  subs?: Link;        // 谁依赖我
  subsTail?: Link;    // 订阅链表尾部
  flags: ReactiveFlags; // 状态标志
}
```

**可视化**:
```
     ┌─────────────┐
     │ReactiveNode │
     ├─────────────┤
deps → Link → Link → ...  (我依赖的)
subs → Link → Link → ...  (依赖我的)
flags: [Mutable][Dirty][Pending]...
     └─────────────┘
```

---

### Link (链接)

**定义**: 连接两个 ReactiveNode 的双向链表节点。

**作用**:
- 建立依赖关系（dep → sub）
- 支持快速遍历
- 版本控制（检测过期依赖）

**结构**:
```typescript
interface Link {
  version: number;              // 版本号
  dep: ReactiveNode;            // 被依赖者
  sub: ReactiveNode;            // 订阅者
  prevSub: Link | undefined;    // 前一个订阅者
  nextSub: Link | undefined;    // 后一个订阅者
  prevDep: Link | undefined;    // 前一个依赖
  nextDep: Link | undefined;    // 后一个依赖
}
```

**双向关系**:
```
从 dep 角度（subs 链表）:
Signal.subs → Link1 → Link2 → Link3
               ↓sub    ↓sub    ↓sub
             Comp1   Comp2   Comp3

从 sub 角度（deps 链表）:
Computed.deps → Link1 → Link2
                ↓dep    ↓dep
              Signal1  Signal2
```

---

### ReactiveFlags (响应式标志)

**定义**: 位运算枚举，表示节点的当前状态。

**标志位**:
```typescript
const enum ReactiveFlags {
  None = 0,           // 000000 - 无状态
  Mutable = 1,        // 000001 - 可变/已变异
  Watching = 2,       // 000010 - 正在监听
  RecursedCheck = 4,  // 000100 - 递归检查中
  Recursed = 8,       // 001000 - 已递归
  Dirty = 16,         // 010000 - 脏数据需更新
  Pending = 32,       // 100000 - 待处理
}
```

**状态含义**:
- **Mutable**: 节点的值可以改变（signals 特有）
- **Watching**: 有活跃的 effect 在监听
- **Dirty**: 值已过时，需要重新计算
- **Pending**: 更新正在传播中

## ⚡ 操作行为术语

### Dependency Tracking (依赖追踪)

**定义**: 当 computed/effect 访问 signal 时，自动建立依赖关系的过程。

**时机**: signal.get() 被调用且 activeSub 存在时

**过程**:
```
1. computed.get() 被调用
2. 设置 activeSub = computed
3. 执行 getter 函数
4. getter 内访问 signal.get()
5. signal 检查 activeSub 存在
6. 调用 link(signal, computed)
7. 创建 Link 节点
8. 添加到双向链表
9. 依赖建立完成
```

---

### Propagation (传播)

**定义**: 当 signal 变化时，沿依赖链向下通知的过程。

**策略**: Push-Pull 混合模型
- **Push 阶段**: 标记所有受影响的节点为 Dirty/Pending
- **Pull 阶段**: 实际访问时才重新计算

**算法**:
```typescript
function propagate(link: Link): void {
  // 非递归，使用栈结构
  let stack: Stack<Link> | undefined;
  
  do {
    const sub = link.sub;
    
    // 标记为 Pending
    sub.flags |= ReactiveFlags.Pending;
    
    // 如果有下级订阅，压栈继续
    if (sub.subs !== undefined) {
      stack = push(stack, sub.subs);
    }
    
    // 移动到下一个或从栈恢复
    link = nextOrPop(stack);
  } while (link !== undefined);
}
```

---

### Flush (刷新)

**定义**: 执行所有待处理的更新操作。

**触发时机**:
- batchDepth === 0 时
- effect 访问 computed 时

**过程**:
```
1. 收集所有 Pending 节点
2. 拓扑排序（确保正确顺序）
3. 依次检查并更新 Dirty 节点
4. 同步 currentValue 和 pendingValue
5. 清除 Pending 标志
```

---

### Batching (批量更新)

**定义**: 合并多次 set 操作为单次更新的机制。

**API**:
```typescript
startBatch();  // batchDepth++
signal1.set(1);
signal2.set(2);
signal3.set(3);
endBatch();    // batchDepth--, if 0 then flush()
```

**优势**:
- 减少不必要的中间更新
- computed 只重新计算一次
- effect 只执行一次

---

## 🏗️ 架构模式术语

### Push-Pull Model (推拉混合模型)

**定义**: 结合推送和拉取优势的响应式更新策略。

**Push (推送)**:
- 数据变化时主动通知
- 优点：及时、精确
- 缺点：可能过度通知

**Pull (拉取)**:
- 需要时才向上查询
- 优点：惰性、按需
- 缺点：可能重复计算

**TypeDOM 的混合方案**:
```
更新流程:
1. signal.set() → Push 阶段开始
2. 沿 subs 链向下传播 Pending 标记 (Push)
3. 标记所有受影响节点为 Dirty
4. effect 执行时访问 computed.get() → Pull 阶段
5. computed 检查 Dirty 标志
6. 如果 Dirty，重新计算并缓存 (Pull)
7. 返回最新结果
```

**对比其他方案**:

| 库 | 模型 | 特点 |
|----|------|------|
| **TypeDOM** | Push-Pull | 惰性 + 精确 |
| MobX | Push | 立即通知所有 |
| Vue | Push-Pull | 类似 TypeDOM |
| Solid | Push | 细粒度推送 |

---

### Lazy Evaluation (惰性求值)

**定义**: 只在真正需要值时才执行计算的优化策略。

**应用场景**:
```typescript
const expensive = computed(() => {
  console.log('Computing...'); // 只在 get() 时执行
  return heavyCalculation();
});

// 不会触发计算
// const val = expensive;

// 触发计算
const val = expensive.get(); // 打印 "Computing..."
```

**优势**:
- 避免不必要的计算
- 提升性能（未使用的 computed 不执行）
- 支持链式计算

---

### Memoization (记忆化)

**定义**: 缓存计算结果，避免重复计算的优化技术。

**实现**:
```typescript
class Computed<T> {
  private _value: T | undefined;
  private _dirty = true;
  
  get(): T {
    if (this._dirty) {
      this._value = this._getter(this._value);
      this._dirty = false; // 标记为干净
    }
    return this._value!;
  }
}
```

**缓存失效条件**:
- 任一依赖变为 Dirty
- 显式调用 markDirty()

## 📈 性能相关术语

### Tree Shaking

**定义**: 构建时移除未使用代码的优化技术。

**TypeDOM 的优势**:
- ES Module 格式
- 无副作用的纯函数
- 模块间无循环依赖

**效果**:
```typescript
// 只使用 signal
import { signal } from '@type-dom/signals';

// Rollup 会移除 computed 和 effect 的代码
// 最终 bundle < 1KB
```

---

### O(1) Dependency Tracking

**定义**: 依赖追踪的时间复杂度为常数级别。

**实现原理**:
- 使用双向链表（Link）
- 直接插入到链表头部/尾部
- 无需遍历查找

**对比**:
- Array: O(n) - 需要查找位置
- Set/Map: O(1) 平均，但有哈希开销
- Link: O(1) 最坏情况也是 O(1)

---

### Non-recursive Propagation

**定义**: 非递归的更新传播算法。

**传统递归方案的问题**:
```typescript
// ❌ 递归可能导致栈溢出
function propagate(node) {
  node.subs.forEach(sub => {
    propagate(sub); // 递归调用
  });
}
```

**TypeDOM 的解决方案**:
```typescript
// ✅ 使用栈结构（堆空间）
function propagate(link: Link): void {
  let stack: Stack<Link> | undefined;
  
  do {
    // 处理当前节点
    // ...
    
    // 压栈而不是递归
    if (hasMoreSubscribers) {
      stack = { value: next, prev: stack };
      continue;
    }
    
    // 从栈恢复
    if (next === undefined && stack) {
      next = stack.value;
      stack = stack.prev;
    }
  } while (true);
}
```

**优势**:
- 不会栈溢出
- 理论上支持无限深度
- 性能更稳定
